# СПЕЦИФІКАЦІЯ ВИМОГ ДО ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ (SRS)
## Pet Marketplace - Платформа для купівлі-продажу та усиновлення домашніх тварин

**Версія документа:** 1.0  
**Дата:** Грудень 2025  
**Статус:** Базова версія  
**Стандарт:** ISO/IEC/IEEE 29148:2011(E)

## 1. ВСТУП

### 1.1 Мета

Метою розробки системи Pet Marketplace є створення комплексної веб-платформи, яка об'єднує функціональність електронної комерції та адміністративної системи управління для ринку домашніх тварин. Система призначена для забезпечення прозорого та безпечного процесу купівлі-продажу тварин, координації адопції через притулки та лікарні для тварин, а також надання освітніх матеріалів про різні породи та догляд за тваринами. Основна мета полягає в створенні екосистеми, яка з'єднує покупців, селекціонерів, притулків та адміністраторів в єдину платформу для забезпечення відповідального та етичного поводження з тваринами.

### 1.2 Обсяг проекту

Проект охоплює розроблення повнофункціональної веб-аплікації, яка складається з двох основних компонентів: клієнтської частини (React.js) та серверної частини (Ruby on Rails API). Система забезпечує управління каталогом тварин, обробку замовлень через платіжну систему Stripe, управління процесом усиновлення, адміністрування ролей користувачів та аналітику активності платформи. Обсяг включає розроблення баз даних PostgreSQL, налаштування безпеки та аутентифікації через JWT токени, а також імплементацію фонових завдань для обробки платежів та відправки сповіщень.

### 1.3 Сфера застосування проекту

Розроблювана система буде застосовуватися в сфері цифрової комерції та комунітарних послуг, специфічно в нішевому сегменті купівлі-продажу домашніх тварин та їхнього усиновлення. Платформа буде доступна через веб-браузер для користувачів різних категорій: звичайних покупців, що шукають домашніх улюбленців або товари для них, селекціонерів, що пропонують чистопородних тварин, притулків та лікарень для тварин, що намагаються знайти новий дім для своїх підопічних, та адміністраторів системи. Цільова аудиторія включає осіб від 18 років з різним рівнем комп'ютерної грамотності та з різних географічних регіонів.

### 1.4 Словник термінів

У контексті цього документа визначаються наступні ключові терміни та поняття. Під терміном «Користувач» розуміється будь-яка особа, яка здійснює аутентифікацію в системі та має доступ до функціональних можливостей відповідно до своєї ролі. «Селекціонер» (Breeder) – це особа або компанія, яка розводить та продає чистопородних домашніх тварин через платформу. «Притулок» (Shelter) – організація, яка пропонує тварин для усиновлення та управляє процесом адопції. «Замовлення» – це商ерцейна операція, в якій користувач купує одну або більше тварин або товарів. «Адопція» – процес офіційного передачі тварини з притулку чи лікарні потенційному власнику через подачу заяви та затвердження. «Платіжний намір» (Payment Intent) – запис про платіж, обробленої через систему Stripe. «JWT Токен» – JSON Web Token, що використовується для аутентифікації та авторизації користувачів. «Аналітика» – збір та аналіз даних про активність користувачів та операції на платформі. «Вебхук» – HTTP зворотний виклик, що надходить від зовнішніх сервісів (наприклад, Stripe) для повідомлення про событи.

### 1.5 Глосарій особливих термінів

До додаткових специфічних термінів належать: «Стан тварини» (Status) – позначка, яка вказує на доступність тварини (available, reserved, sold, adopted, pending, rejected); «Категорія тварини» (Category) – класифікація типу тварини (собаки, коти, птиці, риби, інші); «Порода» (Breed) – конкретна порода в межах категорії; «Роль» (Role) – визначений набір прав та привілеїв користувача (user, breeder, shelter, admin); «Функціональні вимоги» – описи того, що система повинна робити; «Нефункціональні вимоги» – обмеження та характеристики якості; «Сценарій використання» (Use Case) – послідовність дій, які здійснює користувач для досягнення певної мети.

## 2. ЗАГАЛЬНИЙ ОПИС

### 2.1 Класи та характеристики користувачів

#### 2.1.1 Класи користувачів

Система підтримує чотири основні класи користувачів, кожен з яких має свої специфічні функції та можливості. Перший клас – це звичайні користувачі (Regular Users), які є основною масою людей, що використовують платформу для пошуку та покупки домашніх тварин або товарів для них. Вони мають можливість переглядати каталог тварин, фільтрувати їх за різними критеріями, розмісцювати замовлення, управляти своїм профілем та переглядати історію своїх покупок та заяв про адопцію. Другий клас – селекціонери (Breeders), які є зареєстрованими продавцями чистопородних тварин. Вони можуть додавати нові оголошення про тварин, управляти їхніми даними, переглядати замовлення на своїх тварин та комунікувати з потенційними покупцями. Третій клас – організації притулків та лікарень для тварин (Shelters), які управляють процесом адопції. Вони публікують тварин, що потребують нового дому, обробляють заяви про адопцію, схвалюють або відхиляють кандидатів та ведуть облік успішних адопцій. Четвертий клас – адміністратори системи (Administrators), які мають найширший набір прав та можливостей для управління платформою, включаючи модерацію контенту, управління користувачами, налагодження системних параметрів та аналіз даних.

#### 2.1.2 Ролі користувачів

Кожен користувач мусить мати призначену роль, яка визначає його права та доступ до функцій системи. Роль визначається під час реєстрації або змінюється адміністратором. Звичайні користувачі мають базовий доступ до перегляду каталогу, пошуку, створення замовлень та управління своїм профілем. Селекціонери отримують додатковий доступ до панелі керування своїми оголошеннями, можливість завантажувати фотографії тварин та переглядати статистику переглядів. Притулки мають спеціалізовані інструменти для управління заявами про адопцію, включаючи можливість переглядати анкети потенційних власників, залишати коментарі та приймати остаточні рішення. Адміністратори мають доступ до всіх функцій платформи, включаючи можливість видалення контенту, заморожування акаунтів, переведення користувачів в інші ролі та доступ до детальної аналітики системи.

#### 2.1.3 Характеристики користувачів

Цільові користувачі платформи характеризуються різноманітним віком, освітнім рівнем та технічною грамотністю. Переважна більшість користувачів становлять люди віком від 18 до 65 років з базовими навичками роботи з веб-браузерами та соціальними мережами. Багато користувачів не є технічними спеціалістами, тому інтерфейс повинен бути інтуїтивним та простим для використання. Частина користувачів, особливо селекціонери та притулки, матимуть певний рівень технічної підготовки та зможуть працювати з більш складними функціями. Географічне розташування користувачів може бути різноманітним, тому система повинна підтримувати українську та англійську мови. Залежно від ролі користувачів можуть мати різні очікування: звичайні користувачі очікують швидкого та простого пошуку, селекціонери – зручних інструментів для управління своїми оголошеннями, притулки – надійного механізму для управління адопціями.

### 2.2 Робоче середовище

Система розроблена як веб-аплікація, що працює в архітектурі клієнт-сервер. Клієнтська частина виконується в браузері користувача використовуючи технологію React.js версії 19.2.0 та взаємодіє з сервером через REST API. Сервер побудований на основі Ruby on Rails версії 7.1.6 та використовує PostgreSQL як основну СУБД. Система призначена для роботи в середовищах з підтримкою Node.js (для клієнта) та Ruby (для сервера). Клієнт розгортається як статичні файли через веб-сервер і повинен підтримуватися всіма сучасними браузерами (Chrome, Firefox, Safari, Edge). Сервер може бути розгорнений як на локальних серверах, так і на хмарних платформах (AWS, Heroku, Docker контейнерах). Система включає фонові робочі процеси Sidekiq для обробки асинхронних завдань таких як отправка листів та обробка платежів. Важливим аспектом робочого середовища є наявність сервісу обробки платежів Stripe та SMTP сервера для відправки електронних листів користувачам.

### 2.3 Суміжні системи та залежності

Система Pet Marketplace взаємодіє з кількома зовнішніми сервісами та системами. Найважливішою є платіжна система Stripe, яка обробляє платежі користувачів та відправляє вебхуки про результати операцій. Система також повинна мати можливість відправляти електронні листи через SMTP, що використовується для повідомлень про замовлення, скидання пароля та інших критичних подій. Якщо в майбутньому планується експортувати аналітичні дані, система повинна підтримувати інтеграцію з сервісами аналітики. Система залежить від безперебійної роботи PostgreSQL бази даних та Redis для кешування та роботи Sidekiq черг. На клієнтській стороні система залежить від бібліотек React Router для навігації, Axios для HTTP запитів, Chart.js для відображення графіків та React Markdown для рендерення контенту статей.

## 3. ОСОБЛИВОСТІ СИСТЕМИ

### 3.1 Функціональний огляд

Система Pet Marketplace реалізує комплекс функцій, які дозволяють користувачам взаємодіяти з платформою в залежності від їхньої ролі. На найвищому рівні систему можна розділити на кілька основних функціональних модулів: модуль аутентифікації та управління користувачами, модуль каталогу тварин, модуль замовлень та платежів, модуль адопції, модуль контенту (статей та інформації про породи), адміністративна панель та модуль аналітики. Кожен модуль складається з набору взаємопов'язаних функцій, що забезпечують виконання специфічних завдань. Система розроблена таким чином, щоб дозволити легко додавати нові функції в майбутньому та модифікувати існуючі без потреби в повній переробці коду.

### 3.2 Функціональні вимоги

#### 3.2.1 Управління користувачами та аутентифікація

Система повинна надавати можливість новим користувачам реєструватися на платформі, вказуючи їхнє ім'я, прізвище, електронну адресу, номер телефону та пароль. Пароль повинен мати мінімальну довжину в 8 символів та містити принаймні одну прописну букву, одну малу букву, одну цифру та один спеціальний символ. Система повинна перевіряти унікальність електронної адреси та формат номера телефону (українські номери в форматі +380XXXXXXXXX або 0XXXXXXXXX). При реєстрації користувач повинен обрати свою роль: звичайний користувач, селекціонер або притулок. Система повинна надавати функцію входу в акаунт через введення електронної адреси та пароля. При успішному входові система повинна генерувати JWT токен, що використовується для аутентифікації подальших запитів. Токен повинен мати час дії 7 днів, а також генерувати refresh токен з часом дії 30 днів для оновлення основного токена без необхідності повторного входу. Користувач повинен мати можливість вийти з системи, що призводить до аннулювання токена на клієнтській стороні. Система повинна надавати функцію відновлення пароля, при якій користувач може запросити посилання на скидання пароля, яке надсилається на його електронну адресу. Користувач повинен мати можливість переглядати та редагувати свій профіль, включаючи особисту інформацію, адресу доставки та контактні дані. Система повинна зберігати історію входів та зовнішніх виходів для цілей безпеки та аудиту.

#### 3.2.2 Управління каталогом тварин

Система повинна забезпечувати централізований каталог всіх доступних тварин на платформі, до якого мають доступ всі користувачі для перегляду. Селекціонери та притулки повинні мати можливість додавати нові записи про тварин, вказуючи всю релевантну інформацію включаючи ім'я, категорію (собака, кіт, птиця, риба, інше), породу, стать, вік в місяцях, опис, фотографії та статус доступності. Тварини можуть мати один з наступних статусів: available (доступна для покупки/адопції), reserved (зарезервована), sold (продана), adopted (усиновлена), pending (очікує обробки), rejected (відхилена). Для тварин для продажу селекціонери повинні встановлювати ціну, а для тварин для адопції цього не потрібно. Система повинна дозволяти користувачам шукати тварин за різними критеріями: категорія, порода, стать, ціна (мінімум-максимум), вік (мінімум-максимум), селекціонер або притулок. Результати пошуку повинні мати можливість сортування за новизною, ціною (зростання/спадання), віком (зростання/спадання) та назвою. Система повинна виводити останні оновлення до каталогу на головній сторінці та рекомендувати популярних тварин на основі кількості переглядів. Система повинна забезпечувати детальні сторінки для кожної тварини з повною інформацією, фотогалереєю та можливістю залишити запит про покупку/адопцію. Селекціонери та притулки повинні мати можливість редагувати та видаляти інформацію про своїх тварин, але тільки якщо вони не були проданими/усиновленими. Система повинна підтримувати завантаження та збереження фотографій тварин в безпечній локації на серверу.

#### 3.2.3 Управління замовленнями та платежами

Система повинна дозволяти користувачам додавати тварин у кошик та переходити до оформлення замовлення. При оформленні замовлення користувач повинен вказати адресу доставки та переглянути підсумкову вартість замовлення. Система повинна обчислювати загальну вартість замовлення як суму цін всіх тварин/товарів та додатків доставки. Система повинна інтегруватися з платіжною системою Stripe для безпечної обробки платежів. При натисканні на кнопку оплати система повинна перенаправити користувача на безпечну сторінку Stripe для введення деталей кредитної карти. Після успішного платежу Stripe надсилає вебхук підтвердження, на основі якого замовлення позначається як оплачене та статус тварини змінюється на reserved або sold. Користувач повинен отримати електронний лист з підтвердженням замовлення та номером для відстеження. Система повинна дозволяти переглядати історію замовлень у профілі користувача з деталями про кожне замовлення включаючи дату, товари та статус доставки. Система повинна підтримувати скасування або повернення замовлень протягом певного періоду з автоматичним повернення коштів через Stripe. Адміністратори повинні мати можливість переглядати всі замовлення в системі та редагувати їхні статуси для цілей управління та аудиту. Система повинна вести журнал всіх платіжних операцій включаючи успішні платежі, невдалі спроби та повернення грошей.

#### 3.2.4 Управління адопцією

Система повинна надавати спеціалізований процес для управління адопцією тварин з притулків. Притулки повинні публікувати тварин для адопції з детальною інформацією про їхній характер, здоров'я та особливі потреби. Звичайні користувачі повинні мати можливість подавати заяви про адопцію, заповнюючи форму з інформацією про себе та своє живе середовище. Заява повинна включати опис причин, чому користувач хоче усиновити тварину, інформацію про його/її житлові умови, сім'ю та попередній досвід з тваринами. Притулки повинні отримувати повідомлення про нові заяви та мати можливість їх переглядати, залишати коментарі та приймати рішення (затвердження або відхилення). Користувач повинен отримувати сповіщення про рішення притулку щодо його заяви. Якщо заява затверджена, користувач повинен мати можливість планувати дату передачі тварини та отримувати інструкції про необхідні документи. Система повинна дозволяти користувачам переглядати статус своїх заяв в особистому кабінеті та комунікувати з притулками щодо адопції. Користувачі повинні мати можливість переглядати список тварин, доступних для адопції, обраних за притулком або іншими критеріями.

#### 3.2.5 Управління контентом та інформацією про породи

Система повинна мати модуль, що дозволяє авторизованим користувачам (модераторам та адміністраторам) публікувати статті про різні породи тварин, способи їхнього догляду та інші освітні матеріали. Кожна стаття повинна мати заголовок, текст вмісту з підтримкою форматування Markdown, автора та дату публікації. Система повинна категоризувати статті за типом тварини та дозволяти користувачам легко їх знаходити. Користувачи повинні мати можливість переглядати список порід для кожного типу тварини з детальною інформацією про кожну породу включаючи розмір, характер, здоров'я та способи догляду. Система повинна дозволяти модераторам редагувати та видаляти неправильний або застарілий контент. Користувачи повинні мати можливість залишати коментарі та оцінки на статтях для надання зворотного зв'язку автору.

#### 3.2.6 Панель управління селекціонера та притулку

Селекціонери повинні мати доступ до спеціалізованої панелі управління, де вони можуть переглядати та керувати всіма своїми оголошеннями про тварин. Панель повинна показувати кількість переглядів, запитів та замовлень для кожної тварини. Селекціонери повинні мати можливість переглядати деталі замовлень на своїх тварин та комунікувати з покупцями щодо деталей доставки та інших питань. Притулки повинні мати аналогічну панель управління, але з акцентом на управління адопціями замість замовлень. Вони повинні переглядати всі заяви про адопцію для своїх тварин, їхні статуси та деталі потенційних власників. Обидві групи (селекціонери та притулки) повинні отримувати сповіщення про нові замовлення/заяви та важливі оновлення щодо своїх тварин.

#### 3.2.7 Адміністративна панель

Адміністратори повинні мати доступ до повнофункціональної панелі управління системою, через яку вони можуть управляти всіма аспектами платформи. Вони повинні мати можливість переглядати, редагувати та видаляти користувачів, змінювати їхні ролі та статус акаунту (активний, заморожений, видалений). Адміністратори повинні мати можливість модерувати контент включаючи оголошення про тварин та статті для забезпечення дотримання стандартів якості та правил платформи. Вони повинні мати доступ до журналів платежів для перевірки коректності операцій та розслідування проблем. Адміністратори повинні мати можливість переглядати детальні аналітичні дані про активність платформи включаючи кількість користувачів, замовлень, адопцій та доходів. Вони повинні мати можливість налаштовувати системні параметри такі як комісійні збори, повідомлення та інші конфігурації без необхідності зміни коду. Адміністратори повинні мати можливість керувати вебхуками платежів та переглядати журнал всіх вебхуків для цілей налагодження. Вони повинні мати можливість отправляти масові сповіщення користувачам та селекціонерам.

#### 3.2.8 Аналітика та звітність

Система повинна збирати дані про активність користувачів включаючи кількість відвідувань, переглядів тварин, замовлень та адопцій. Дані повинні периодично обробляватися фоновими завданнями та зберігатися в базі даних для наступного аналізу. Система повинна надавати адміністраторам детальні звіти про яких показниках діяльності платформи: загальна кількість користувачів, активні користувачі, дохід від продажів, кількість замовлень, успішні адопції та інші метрики. Звіти повинні мати можливість фільтрування за часовими періодами та категоріями. Система повинна генерувати таблиці та графіки для візуалізації даних, що полегшує розуміння трендів. Система повинна інформувати адміністраторів про важливі подій такі як проблеми з платежами, невдалі операції та аномальна активність.

## 4. ВИМОГИ ДО ЗОВНІШНЬОГО ІНТЕРФЕЙСУ

### 4.1 Користувацький інтерфейс

#### 4.1.1 Загальні принципи дизайну

Користувацький інтерфейс системи повинен бути розроблений з дотриманням принципів зручності використання, доступності та естетичної привабливості. Інтерфейс повинен бути адаптивним та коректно відображатися на пристроях різних розмірів: настільні комп'ютери, планшети та мобільні телефони. Текст повинен бути легко читаним з використанням шрифтів достатнього розміру та контрастури. Кольорова схема повинна бути узгодженою та зручною для очей, з чіткою різницею між активними та неактивними елементами. Навігація повинна бути інтуїтивною, з чіткою ієрархією сторінок та логічним розташуванням меню.

#### 4.1.2 Головна сторінка

Головна сторінка повинна містити привітальне зображення або слайдер з основною інформацією про платформу. Вона повинна демонструвати популярних тварин на основі кількості переглядів та недавно доданих оголошень. На головній сторінці повинна бути видна панель швидкого пошуку, що дозволяє користувачам вводити критерії пошуку (категорія, діапазон ціни, район тощо) та переходити до результатів. Сторінка повинна містити посилання на основні розділи системи: перегляд собак, котів, інших тварин, притулків, статей та адміністративної панелі (для адміністраторів). Головна сторінка повинна показувати статистику платформи (кількість тварин, успішних адопцій тощо) для підвищення довіри користувачів. Вона повинна містити інформацію про переваги платформи та відповіді на часто задавані питання.

#### 4.1.3 Сторінка каталогу тварин

Сторінка каталогу повинна виводити список тварин в умовно розділеній на карточки форматі, де кожна карточка показує фото, ім'я, породу, стать, вік та ціну (для продажу) або надпис "Для адопції". На сторінці повинна бути бічна панель фільтрів, що дозволяє користувачам уточнювати результати пошуку за категорією, породою, статтю, цінолю, віком, селекціонером/притулком та статусом. Результати повинні мати можливість сортування за новизною, ціною, віком та назвою. Сторінка повинна підтримувати пагінацію або нескінченний скрол для легкого перегляду великої кількості результатів. При натисканні на карточку тварини користувач переходить на детальну сторінку.

#### 4.1.4 Детальна сторінка тварини

Детальна сторінка повинна показувати всю інформацію про тварину включаючи фотогалерею, опис, категорію, породу, стать, вік, ціну (якщо застосовується), селекціонера/притулку та контактну інформацію. На сторінці повинна бути кнопка "Додати в кошик" для продажі або "Подати заяву про адопцію" для адопції. Сторінка повинна показувати статус тварини та час публікації оголошення. Користувачі повинні мати можливість залишати коментарі та питати питання про тварину. Якщо користувач є авторизованим селекціонером/притулком, він повинен мати можливість редагувати інформацію про тварину.

#### 4.1.5 Кошик та оформлення замовлення

Сторінка кошика повинна показувати список всіх доданих товарів з фото, описом та ціною. Користувач повинен мати можливість змінити кількість або видалити товари з кошика. Сторінка повинна показувати підсумкову вартість та дозволяти користувачу ввести промокод для отримання знижки. При натисканні на кнопку "Оплатити" користувач переходить на сторінку оформлення замовлення, де вводить адресу доставки та переглядає остаточну вартість. Після підтвердження користувач перенаправляється на безпечну сторінку Stripe для введення даних платіжного засобу. Після успішного платежу користувач отримує сторінку підтвердження з номером замовлення та контактною інформацією для зв'язку.

#### 4.1.6 Профіль користувача та історія замовлень

Сторінка профілю повинна дозволяти користувачам переглядати та редагувати особисту інформацію (ім'я, прізвище, номер телефону, адреси доставки). Користувачи повинні мати можливість змінити пароль та керувати параметрами безпеки акаунту. На сторінці повинні бути видні замовлення користувача з статусами доставки та посиланнями для контакту з продавцями. Якщо користувач подавав заяви про адопцію, на сторінці повинні бути видні ці заяви з статусами розгляду. Користувачи повинні мати можливість переглядати історію своїх діяльностей на платформі.

#### 4.1.7 Форма для адопції

Форма для адопції повинна мати поля для введення інформації про користувача (ім'я, адреса, номер телефону), інформацію про житлові умови (тип житла, розмір, наявність двору), інформацію про сім'ю (склад, інші тварини) та мотивацію для адопції. Форма повинна мати зрозумілі інструкції та підказки для кожного поля. Вона повинна валідувати введені дані перед надсиланням. Після успішного надсилання користувач повинен отримати сповіщення про отримання заяви та очікуваний час розгляду.

#### 4.1.8 Панель управління селекціонера/притулку

Панель повинна мати вкладки для перегляду оголошень про тварин, замовлень/заяв, статистики та налаштувань. На вкладці оголошень повинні бути видні всі оголошення з можливістю додавання нових, редагування існуючих та видалення. Для кожного оголошення повинна бути видна статистика переглядів та замовлень. На вкладці замовлень/заяв повинні бути видні всі замовлення/заяви з можливістю зміни статусу та комунікації з користувачами. Панель повинна мати сповіщення про нові замовлення та заяви.

#### 4.1.9 Адміністративна панель

Адміністративна панель повинна мати основні вкладки: користувачі, оголошення, замовлення, адопції, контент, статистика та налаштування. На вкладці користувачів адміністратор повинен переглядати список всіх користувачів, змінювати їхні ролі та статуси. На вкладці оголошень адміністратор повинен модерувати оголошення про тварин. На вкладці замовлень адміністратор повинен переглядати всі замовлення та їхні статуси платежів. На вкладці статистики повинні бути графіки та таблиці з основними показниками діяльності. На вкладці налаштувань адміністратор повинен змінювати конфігурацію системи.

### 4.2 Системні інтерфейси

#### 4.2.1 REST API

Система повинна надавати повнофункціональний REST API, який дозволяє клієнтські та сторонні додатки взаємодіяти з сервером. API повинен дотримуватися REST принципів з використанням стандартних HTTP методів (GET, POST, PUT, DELETE). Всі API запити повинні потребувати валідного JWT токена в заголовку Authorization для аутентифікації. Сервер повинен повертати результати в форматі JSON з відповідними кодами статусу HTTP. API повинен надавати детальні повідомлення про помилки для полегшення налагодження. API повинен обмежувати частоту запитів для запобігання зловживанню та DDoS атакам.

#### 4.2.2 Вебхуки платіжної системи

Система повинна мати кінцевик для отримання вебхуків від Stripe про результати платіжних операцій. Вебхуки повинні включати подій: payment_intent.succeeded, payment_intent.payment_failed, charge.refunded. При отриманні вебхука про успішний платіж система повинна оновити статус замовлення та позначити його як оплачене. При отриманні вебхука про невдалий платіж система повинна оновити статус замовлення та сповістити користувача. Система повинна зберігати записи всіх отриманих вебхуків для аудиту та налагодження.

## 5. НЕФУНКЦІОНАЛЬНІ ВИМОГИ

### 5.1 Вимоги безпеки

Система повинна реалізувати надійний механізм аутентифікації на основі JWT токенів з обов'язковим завантаженням токенів з HTTPS з безпечним браузерним сховищем. Всі критичні операції такі як платежі та видалення користувачів повинні вимагати підтвердження користувача. Система повинна захищати паролі користувачів за допомогою bcrypt хешування з факультативною солю та з мінімальною вартістю bcrypt 12. Всі передачи даних повинні використовувати HTTPS для зашифрування даних в дорозі. Система повинна імплементувати захист від CSRF атак за допомогою токенів. База даних повинна бути захищена від SQL інєкцій через використання параметризованих запитів та ORM (ActiveRecord). Система повинна регулярно оновлювати залежності для закриття відомих уразливостей. Чутлива інформація (паролі, платіжні дані) не повинна логуватися або передаватися в невиконаному видянні. Система повинна реалізувати логування та моніторинг для виявлення підозрілої активності та можливих атак. Адміністратори повинні регулярно переглядати логи безпеки.

### 5.2 Вимоги до продуктивності

Система повинна обробляти як мінімум 100 одночасних користувачів без помітного погіршення продуктивності. Час відповіді для сторінок повинен бути менше 2 секунд на нормальній мережі для перших запитів та менше 500 мс для кешованих запитів. Система повинна обробляти як мінімум 95% платіжних транзакцій менше ніж за 3 секунди від подання до підтвердження. База даних повинна мати можливість обробляти мільйони записів без помітного погіршення продуктивності. Сервер повинен працювати на Linux серверах та мати можливість розгортання в Docker контейнерах та хмарних платформах. Система повинна бути сумісна з PostgreSQL 12 та новішими версіями. Система повинна підтримувати інтеграцію з популярними платіжними системами включаючи Stripe.

Код повинен мати хорошу документацію включаючи архітектурні діаграми, опис функцій та приклади використання. Система повинна мати чітку структуру для добавлення нових функцій та модифікації існуючих без повної переробки коду.

Система повинна підтримувати українську та англійську мови зі можливістю легкого добавлення нових мов в майбутньому. Всі повідомлення користувачу повинні бути перекладені на обидві мови. Система повинна коректно відображати формати дат, часу та валют відповідно до вибраної мови та регіону. Контент такий як назви тварин та описи можуть залишатися в оригінальній мові.

## 6. ПРИПУЩЕННЯ І ЗАЛЕЖНОСТІ

### 6.1 Припущення

Система припускає, що користувачі мають доступ в Інтернет та можуть отримати доступ до веб-браузера. Припускається, що користувачі є добросовісними та не будуть намагатися використовувати систему для незаконних цілей. Припускається, що SMTP сервер для відправки листів буде доступний та налаштований правильно. Припускається, що платіжна система Stripe буде доступна та працюватиме без перебоїв. Припускається, що база даних PostgreSQL буде встановлена та запущена на серверу. Припускається, що сертифікати SSL/TLS будуть встановлені для HTTPS з'єднання.

### 6.2 Залежності

Система залежить від наявності та коректної роботи PostgreSQL бази даних версії 12 або новішої. Система залежить від платіжної системи Stripe для обробки платежів та отримання вебхуків. Система залежить від SMTP сервера для відправки електронних листів користувачам. Система залежить від Ruby версії 3.0.0 та Rails версії 7.1.6 для роботи сервера. Система залежить від Node.js та React для роботи клієнтської частини. Система залежить від Sidekiq та Redis для роботи фонових завдань. Система залежить від відповідної конфігурації CORS на сервері для дозволу запитів з клієнтської частини. Система залежить від наявності інтернет-з'єднання для отримання зовнішніх ресурсів та API запитів.
